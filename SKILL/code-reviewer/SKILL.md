# 前端代码审查规范

## 核心原则：找真正的问题，不制造噪音
审查的目的是发现**会影响用户或维护者**的问题，不是炫技。每个指出的问题必须有明确的影响说明和改进方案。跳过主观风格偏好（除非违反项目规范）。

---

## 🔍 审查维度与权重

| 维度 | 权重 | 核心关注点 |
|------|------|-----------|
| 语义化 & SEO | ⭐⭐⭐⭐ | HTML 结构是否合理，heading 层级是否连贯，是否有 `main`/`nav` 等 landmark |
| 可访问性 (a11y) | ⭐⭐⭐⭐⭐ | 键盘操作、屏幕阅读器支持、颜色对比度、焦点管理 |
| 性能 | ⭐⭐⭐⭐ | 不必要的渲染、大体积依赖、图片优化缺失、内存泄漏 |
| 安全性 | ⭐⭐⭐⭐⭐ | XSS、敏感信息暴露、不安全的第三方依赖引用 |
| 可维护性 | ⭐⭐⭐ | 命名一致性、职责单一、魔数的消除、条件逻辑的可读性 |

---

## 📋 审查清单

### 一、HTML 语义化

**必须指出的问题：**
- `<div>` 滥用 —— 应该用 `<header>`、`<nav>`、`<main>`、`<section>`、`<article>`、`<aside>`、`<footer>` 的地方却用了 `<div>`
- Heading 层级跳跃 —— `<h1>` → `<h3>` 跳过 `<h2>`，或一个页面有多个 `<h1>`
- 列表内容没有用 `<ul>/<ol>/<dl>`，而是用 `<div>` 拼凑
- 表格数据没有用 `<table>`，调用了 `<div>` 模拟
- 链接用 `<div onclick="...">` 而不是 `<a href="...">`
- 按钮用 `<a href="#">` 或 `<div>` 而不是 `<button>`

**正确示例：**
```html
<!-- ❌ 错误 -->
<div class="header"><div class="nav"><div onclick="go('/')">首页</div></div></div>

<!-- ✅ 正确 -->
<header><nav><a href="/">首页</a></nav></header>
```

### 二、可访问性 (a11y)

**必须指出的问题：**

| 问题类型 | 检查方法 | 严重程度 |
|---------|---------|---------|
| 图片缺少 `alt` | 所有 `<img>` 都有 `alt`（装饰图可留空） | 🔴 高 |
| 图标按钮无标签 | `<button>` 只有 SVG 图标时，必须有 `aria-label` | 🔴 高 |
| 表单无 label 关联 | `<input>` 必须有对应的 `<label>` 并关联 `for`/`id` | 🔴 高 |
| 颜色对比度不足 | 文字与背景对比度 < 4.5:1（正文）或 < 3:1（大标题） | 🟡 中 |
| 焦点样式被移除 | `outline: none` 没有替代方案（`:focus-visible`） | 🔴 高 |
| `tabindex` 滥用 | `tabindex="0"` 用在非交互元素，或正值 tabindex | 🟡 中 |
| 自定义组件缺少 ARIA | 非原生组件（如 select 模拟）缺少 `role`/`aria-*` | 🔴 高 |
| 颜色唯一传信息 | 只用颜色表达状态（如红/绿标签），没有额外图标或文字 | 🟡 中 |

**焦点管理专项检查：**
- 弹窗打开后，焦点是否进入弹窗（`focus-trap`）
- 弹窗关闭后，焦点是否回到触发元素
- 路由切换后，焦点是否移到新页面内容（或标题被读屏器读出）

### 三、性能

**必须指出的问题：**

| 问题 | 识别方式 | 改进方向 |
|------|---------|---------|
| 组件内定义不变常量 | `const opts = [...]` 写在组件函数体内 | 移到组件外部 |
| 派生状态可缓存 | 每次渲染都重新计算的值 | 用 `useMemo` / `computed` |
| 内联函数导致的子组件重渲染 | `onClick={() => fn()}` | 用 `useCallback` 或提取 |
| 大列表无虚拟化 | 一次性渲染超过 100 项的列表 | 加虚拟滚动或分页 |
| 图片未优化 | `<img>` 直接引用 2000px 宽原图 | 用 `srcset`/`sizes` 或 CDN 裁剪 |
| 未代码分割 | 200KB+ 的单 JS bundle | `lazy()` + `Suspense` |
| CSS 动画用 `height/width` | 展开/折叠动画操作几何属性 | 改用 `transform: scaleY` 或 FLIP |
| 未清理副作用 | `useEffect` 有订阅/定时器但无 `return` 清理函数 | 补充清理逻辑 |

### 四、安全性

**必须指出的问题（不允许协商）：**

- **XSS**：`dangerouslySetInnerHTML` 或 `v-html` 的内容没有经过 DOMPurify 等清洗
- **敏感信息**：API key、token、密码出现在前端代码或 `.env` 中（应只在后端）
- **不安全链接**：`<a target="_blank">` 没有 `rel="noopener noreferrer"`
- **eval/Function 执行用户输入**：永远不能用
- **依赖安全**：引用了已知漏洞版本的 npm 包（可提示检查 `npm audit`）

### 五、可维护性

**应指出但非阻塞的问题：**

- **命名不一致**：同一概念用了不同名称（如 `isActive` 和 `isSelected` 混用）
- **魔数**：出现含义不清的数字（如 `if (count > 5)`，5 是什么意思？）
- **过深的嵌套**：JSX/模板缩进超过 4 层，应拆分子组件
- **重复代码**：同一逻辑在两处以上完全相同，应提取
- **职责混乱**：一个组件同时处理数据获取、业务逻辑、UI 渲染
- **条件地狱**：`if/else` 嵌套超过 3 层的，用 early return 或 lookup map 重构
- **注释与代码不一致**：注释说 A 但代码做 B（比没注释更危险）

## 📝 审查输出格式

每次审查按以下结构输出：

```
## Code Review: <文件名或组件名>

### 🔴 必须修复 (Blockers)
> 影响功能、安全或 a11y 合规的问题

1. [问题描述]
   - 位置：<文件名:Lxx-Lxx>
   - 影响：<说明用户/维护者会受到什么具体影响>
   - 修复：<具体的修改方案>

### 🟡 建议优化 (Recommendations)
> 提升质量但不阻塞发布

1. [建议描述]
   - 位置：<文件名:Lxx-Lxx>
   - 理由：<为什么建议这样做>
   - 方案：<具体的优化方向>

### ✅ 做得好的地方
> 值得保持的做法

1. <具体说明哪里做得好及其原因>
```

## 🚫 不审查的内容

以下内容除非用户特别要求，否则不应出现在审查中：
- 代码风格偏好（缩进用 2 格还是 4 格、单引号还是双引号）—— 让 linter 管
- 架构选型争议（Vue 还是 React、Redux 还是 Zustand）—— 已经选了就尊重
- 命名风格（驼峰 vs 下划线）—— 只关注一致性，不关注偏好
- 单行代码的"更优雅写法"—— 如果功能正确且可读，不需要改
